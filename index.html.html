Section 01
What is Greykite?
Greykite is LinkedIn's production-grade forecasting framework, open-sourced in 2021. It addresses the fundamental weakness of statistical anomaly methods: they don't understand time.

🏗 Core Philosophy

Greykite treats anomaly detection as a two-step problem: (1) Build the best possible forecast of what a metric "should" be given its historical patterns. (2) Compare what actually happened against that forecast — anything outside the model's confidence interval is an anomaly. This is fundamentally different from IQR or rolling z-scores which have no temporal awareness.

⚙️ Two Algorithms Inside

Greykite ships with two forecast backends: Silverkite (LinkedIn's proprietary algorithm — fast, interpretable, production-proven at scale) and Prophet (Facebook's additive model). The AUTO template benchmarks both and picks the winner. Silverkite is the default champion — it's 10–100× faster than Prophet on large datasets.

📐 Unified Framework

Unlike running Prophet or SARIMA standalone, Greykite adds a full anomaly detection layer on top of any forecast model. The ADConfig + GreykiteDetector wrapping handles confidence interval tuning, alert rate calibration, and outlier-aware retraining — none of which Prophet exposes natively.

🏭 Production Origin

Greykite was designed for LinkedIn's business metrics infrastructure — monitoring millions of KPIs daily across user engagement, ads revenue, and job postings. It's built for batch processing at scale, irregular time series, sparse data, and campaigns with short history — exactly the challenges we face with TOFU campaign metrics.

Section 02
Silverkite — The Engine Under the Hood
Silverkite is Greykite's primary forecasting algorithm. Understanding it is key to understanding why Greykite detects anomalies correctly while IQR doesn't.
📈

Additive Decomposition

Silverkite decomposes a time series into: Trend (long-run direction), Seasonality (weekly, monthly, yearly cycles), Holiday Effects (named event regressors), and Residual (what's left). An anomaly is only in the residual — not in the expected seasonal swing.

Core Model
⚡

Ridge Regression Backend

Unlike Prophet which uses MCMC (slow), Silverkite fits a regularized linear model (Ridge) on engineered features. The features include Fourier terms for seasonality, piecewise linear trend, and one-hot holiday indicators. This makes it 10–100× faster than Prophet with comparable accuracy.

Speed
📅

Fourier-Based Seasonality

Seasonality is captured using Fourier series — pairs of sin/cos terms at different frequencies. Weekly seasonality uses 3 Fourier pairs (period 7 days), yearly uses 8–15 pairs. This gives smooth, flexible seasonal curves without overfitting, unlike simple day-of-week dummies.

Feature Engineering
🗓

Holiday Regressors

For each US holiday, Silverkite creates a binary regressor for the holiday day plus pre_num_days and post_num_days windows. Each gets its own fitted coefficient — so Black Friday pre-window can have a large positive coefficient while the post-Thanksgiving slump has a negative one.

Context-Aware
🔀

Changepoint Detection

Silverkite uses piecewise linear trend with automatic changepoint detection. When a campaign fundamentally changes its trajectory (new budget, channel shift), the model adapts its trend segment rather than treating the new level as an ongoing anomaly — avoiding the "level shift" false positive problem.

Trend
🎯

AUTO Template Logic

When ModelTemplateEnum.AUTO is used, Greykite evaluates multiple configurations: Silverkite-fast, Silverkite-full, and Prophet. It runs cross-validation on each and selects the one with the lowest prediction error. For campaign KPIs, Silverkite-fast typically wins on the speed-accuracy tradeoff.

AutoML
Key Insight: The reason Greykite catches real anomalies and ignores seasonal patterns is that Silverkite explicitly models the seasonality before computing residuals. An IQR model sees a Black Friday spike as "extreme value." Greykite sees it as "expected large positive residual for this holiday regressor" — so it doesn't flag it. Only a drop that falls outside the expected holiday distribution is flagged.

Section 04
ADConfig — The Detection Policy
ADConfig is what separates Greykite from "just running Prophet." It adds a feedback-loop calibration layer that automatically adjusts the model's confidence interval until the desired alert rate is achieved.
01
Fit Forecast
Silverkite/Prophet fits on full history. Learns trend, seasonality, holiday effects.
02
Generate PI
Prediction interval [lower, upper] generated at initial coverage=0.95 setting.
03
Evaluate Rate
Count what % of historical points fall outside the interval. Compare to target 0.2%.
04
Tune CI Width
Reward function widens or narrows the interval until alert rate matches target. Iterative optimization.
05
Flag Anomalies
Final calibrated interval applied. Points outside → is_anomaly_predicted = True.

The Reward class in Greykite defines the optimization objective for CI calibration. Instead of a fixed z-score threshold, it maximizes a reward function balancing coverage and precision. This is the mechanism behind target_anomaly_percent.

ad_config.py — Home Depot implementation
from greykite.detection.detector.config import ADConfig
from greykite.detection.detector.reward import Reward
from greykite.detection.detector.greykite import GreykiteDetector

# target_anomaly_percent: ADConfig tunes CI width
# so ~0.2% of historical days are flagged.
# Lower = more strict (fewer alerts)
# Higher = more sensitive (more alerts)

    target_anomaly_percent=0.2
)

detector = GreykiteDetector(
    forecast_config=forecast_config,
    ad_config=ad_config

# Fit learns the expected behavior

greykite_pipeline.py — full per-series logic
def process_campaign_group(group, kpis=["total_clicks", "total_impressions"]):
    for kpi in kpis:
        # ── 1. Prep series ──────────────────────────────────────────
        gk_data = group[['Date', kpi]].rename(
            columns={"Date": "ts", kpi: "y"}
        )
        gk_data['ts'] = pd.to_datetime(gk_data['ts'])
        gk_data['y']  = gk_data['y'].fillna(0).astype(float).ffill()

        # ── 2. Quality gate ─────────────────────────────────────────
        # Skip constant series, zero-inflated (>90%), <30 days
        if is_problematic_series(group, kpi): continue
        if len(gk_data) < 30: continue

        # ── 3. Config ────────────────────────────────────────────────
        forecast_config = ForecastConfig(
            model_template=ModelTemplateEnum.AUTO.name,
            metadata_param=MetadataParam(time_col="ts", value_col="y", freq="D"),
            coverage=0.95,
            forecast_horizon=1,
            evaluation_period_param=EvaluationPeriodParam(
                test_horizon=7, cv_max_splits=0
            ),
            model_components_param=ModelComponentsParam(events={
                "holiday_lookup_countries": ["US"],
                "holiday_pre_num_days": 2,
                "holiday_post_num_days": 2,
            })
        )
        ad_config = ADConfig(target_anomaly_percent=0.2)

        # ── 4. Fit + Predict ─────────────────────────────────────────
        detector = GreykiteDetector(
            forecast_config=forecast_config,
            ad_config=ad_config
        )
        detector.fit(data=DetectorData(df=gk_data))
        result = detector.predict(DetectorData(df=gk_data))
        pred   = result.pred_df

        # ── 5. Extract + MAPE ────────────────────────────────────────
        mape = calculate_mape(pred["actual"].values,
                               pred["forecast"].values)

        # ── 6. Collect results ───────────────────────────────────────
        rows = pred[["ts", "actual", "forecast",
                     "forecast_upper", "forecast_lower",
                     "is_anomaly_predicted"]].copy()
        rows["campaign_id"] = group["campaign_id"].iloc[0]
        rows["KPI"]         = kpi

# Parallel execution across all campaign groups
with mp.Pool(processes=min(100, mp.cpu_count())) as pool:
    results = pool.starmap(
        process_campaign_group,
        [(group, kpis) for _, group in df.groupby("campaign_id")]
    )

Greykite Output · How to read the pred_df
How to use MAPE for trust scoring: Before acting on a Greykite_Anomaly = TRUE, check the corresponding row in MAPE_TOFU_Weekly_Updates. A series with MAPE > 100% (common in sparse SEM campaigns with zero-dominated history) means the forecast was unreliable — the anomaly flag may be noise. Filter to MAPE < 50% for high-confidence alerts. This is a key advantage over IQR, which has no equivalent fit-quality diagnostic.
